# Guiding Questions

## README File

**What would new users need to do to get started?**
New users need to know what exactly the repos function is. What does it do? What problems does it solve? Moreover users would like to know the tech stack. An extremly important part of the readme is how to actually install and use the repo. Finally a user may like to know what recent changes are and who has been working on the project.

**What sections will need to be updated once development begins?**
All sections will likely need to be updated eventually

---

## Pull Request Template

**What information would help reviewers quickly understand the purpose and impact of the PR?**
A description of what is included in the PR. What kind of changes the PR makes. If there are related open issues.

**What common mistakes or missing elements do we want to help contributors avoid?**
We want to ensure they are inserting all relevant information. Often times contibutors don't want to type out a bunch of text about their change as it is tedious but it is important.

---

## Issue Template

**What types of issues do we expect users to report?**
Bugs, optimizations, fixes, and feature requests.

**What information would help us fix bugs or evaluate feature requests faster?**
Stuff like a description of what we want to be happening, what the behaviour is now, how importantthis change would be, how much time it would take.

## Contribution Guide

**What basic information would a new contributor need to know about our project to get started?**
They need to know what the standards they are expected to meet as well as how they should behave towards their fellow contributors

**What communication and collaboration standards do we need to promote effective contribution?**
Empathy and kindness. Also foster an environment that promotes communication and does not shame others for their ideas.

---

## Support Page

**Where should users go if they have a question about using our project?**
Start with the README for basic usage. For broader questions or help, use GitHub Discussions. For bugs or specific problems, open an issue.

**What types of questions should go in Discussions vs. Issues?**
Discussions are for general questions, ideas, and help requests. Issues are for actionable items like bug reports and feature requests.

---

## Choosing a License

**Under what circumstances do we want people to be able to use or modify our source code?**
We want anyone to freely use, modify, and distribute our code as long as they include attribution. We chose the MIT License for its simplicity and permissiveness.

**What is meant by a 'copyleft' license, and what are the implications for commercial software?**
A copyleft licenserequires that any modified or derivative version of the code also be released under the same open source license.
